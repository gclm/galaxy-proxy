# 项目愿景与目标

## 一句话定义

Galaxy Router 是一个 **AI 协议互转代理网关**，让不同协议的 AI 客户端能够透明访问不同协议的上游服务。

## 要解决的问题

| 问题 | 现状 | 目标 |
|------|------|------|
| 协议不互通 | OpenAI SDK 只能调 OpenAI，Anthropic SDK 只能调 Anthropic | 任意客户端调任意上游 |
| 多 Key 管理混乱 | 每个服务单独配置 Key | 统一管理，按组分配 |
| 无法负载均衡 | 单点故障，手动切换 | 自动故障转移 + 加权分配 |
| 缺少用量统计 | 不知道谁用了多少 | 按 Key/模型/时间统计 Token 和成本 |

## 目标用户

- **个人开发者**：有多个 AI 服务 Key，想统一管理和切换
- **小团队**：需要共享 AI 服务配额，按成员统计用量
- **AI 应用开发者**：需要在不同协议间做兼容适配

## 核心价值

```
客户端 (任意协议) → Galaxy Router → 上游服务 (任意协议)
                      ↓
                   统计 + 负载均衡 + 故障转移
```

## 参考项目定位

| 项目 | 定位 | 我们借鉴什么 |
|------|------|-------------|
| Octopus | 个人 LLM 聚合 | 分组配置的简单易用 |
| Sub2API | 商业级 API 分发 | 负载均衡策略 + 统计维度 |
| CCG Gateway | 桌面端 CLI 工具 | Rust 架构 + 流式处理 |

## 不做什么（边界）

- ❌ 不做多用户系统（单用户模式，首次启动初始化管理员）
- ❌ 不做支付系统（Sub2API 的商业化功能）
- ❌ 不做桌面 GUI（纯 CLI / 服务端部署）
- ❌ 不做 Gemini 协议（第一版只做 OpenAI + Anthropic）
- ❌ 不做排队机制（第一版直接返回 429，后期可选开启）
- ❌ 不做多实例部署（单机 SQLite，进程重启后内存状态丢失）

## Web 管理面板（初版）

初版只做简单的管理功能，前端用 React：

| 功能 | 说明 |
|------|------|
| 渠道管理 | 增删改查渠道、API Key、模型映射 |
| 分组管理 | 增删改查分组、负载均衡模式、渠道项 |
| 客户端 API Key | 生成/吊销客户端 API Key |
| 设置 | 基础配置查看（TOML 只读） |
| 统计概览 | 用量统计、成本查看 |
